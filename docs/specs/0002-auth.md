# 0002 — Accounts & Authentication

**Status:** Done
**Phase:** 0
**Platform:** Both
**Implements:** PRD §4, TRD §4, §6  ·  **Depends on:** 0001 (foundations)
**Date:** 2026-07-09

---

## 1. Summary
Email/password accounts so a listener's progress and favourites follow them across devices (PRD §4).
This spec covers the four `(auth)` screens — **login, register, forgot-password, update-password** —
the `authStore` session lifecycle, the English→Kazakh auth-error mapping, and the server-side
middleware redirects. Profiles are provisioned automatically by the `handle_new_user` trigger (TRD §6).

## 2. Goals / non-goals
- **Goals:** register (name + email + password), sign in, request a reset email, set a new password;
  localised errors; automatic `profiles` provisioning; session bootstrap + reactive auth state.
- **Non-goals:** OAuth/social login, email OTP/magic-link, phone auth, profile editing (settings spec).
  Anonymous browsing is redirected to sign-in, not supported in-app (PRD §3).

## 3. User stories / behavior
Per PRD §4 acceptance criteria: a new user registers, is signed in, and lands on `/home`; forgot-password
sends a reset link that opens `/update-password`; wrong-password / duplicate-email / weak-password each
show a **specific Kazakh message**. Signed-out users on `(app)` routes are sent to `/login` (with a
`redirect` back); signed-in users on `/login`/`/register` are sent to `/home` — enforced in `proxy.ts`,
not just the UI.

## 4. Design
- **Screens** (`src/app/(auth)/*`, all `"use client"`): each renders through the shared `AuthPanel`
  (`components/auth/AuthPanel.tsx`) and its exported input/submit class constants. Forms use `noValidate`,
  `useId()` field ids, a password show/hide toggle, and a success panel (`role="status"`). `register`
  and `update-password` enforce a client-side 6-char minimum before calling the store.
- **`authStore`** (`store/authStore.ts`, TRD §13): `signIn` → `signInWithPassword`; `signUp` →
  `signUp({ options: { data: { full_name } } })`; `resetPassword` → `resetPasswordForEmail(email,
  { redirectTo: origin + "/update-password" })`; `updatePassword` → `updateUser({ password })`;
  `signOut` → `signOut()` then hard-navigate to `/login`. `initialize()` subscribes to
  `onAuthStateChange` and calls `getSession()`; `applySession` sets a fallback user from auth metadata,
  then hydrates the `profiles` row in the background; `TOKEN_REFRESHED`/same-email `USER_UPDATED` skip
  the re-fetch to avoid needless work.
- **Bootstrap:** `AuthProvider` (`components/layout/AuthProvider.tsx`, mounted in the root layout)
  calls `initialize()` once on mount.
- **Error mapping:** `lib/authTranslations.ts` `translateAuthError()` matches GoTrue English signatures
  (invalid credentials, already-registered, weak/short password, bad email, unconfirmed, rate limit,
  network, session expired, signups disabled…) to Kazakh; unmatched input is returned unchanged.

## 5. Data & interface changes
- **Data model:** no schema change — reuses `profiles` from spec 0001. On `auth.users` insert, the
  `handle_new_user()` `SECURITY DEFINER` trigger writes `profiles(id, email, full_name)` (bypassing RLS,
  idempotent via the `unique_violation` guard). The `authStore` also has a client-side
  `getOrCreateProfile` fallback that inserts under the `profiles_insert` RLS policy (`auth.uid() = id`)
  if the row is absent — covering a trigger race (TRD §6).
- **Interfaces:** the `authStore` action surface above; `translateAuthError`; `proxy.ts` redirect
  behaviour (`PROTECTED_PATHS`, `AUTH_PATHS`, sanitised `redirect`).

## 6. Accessibility
Each form field is `<label htmlFor>`-bound to a `useId()` input, marked `required` + `aria-required`;
leading icons are `aria-hidden`. On error, inputs flip `aria-invalid` and point `aria-describedby` at an
error region with `role="alert"` + `aria-live="assertive"`, so the Kazakh message is announced; the
register/update password field keeps its hint id in `aria-describedby` alongside the error. The
password toggle is a real `<button>` with a Kazakh `aria-label`, `aria-pressed`, and `aria-controls`.
Submit buttons expose `aria-busy` while loading, and the `Loader2` spinner carries
`motion-reduce:animate-none`. Success panels use `role="status"` + `aria-live="polite"`. Each page has
exactly one `h1` (the `AuthPanel` title), visible `focus-visible` rings, secondary text on the AA muted
token (**`#6B6B6B` / jaryq-text-secondary**), and reduced-motion honoured via `motion-reduce:*`.
Verified by the axe-core suite (PRD §14, TRD §9). Errors are **never** shown as raw English or a code.

## 7. Edge cases & failure handling
- Wrong password / duplicate email / weak password → specific Kazakh strings; unmatched GoTrue errors
  fall through to the original message rather than a blank state.
- Network failure → `"Желілік байланыс қатесі…"`; rate-limit → the throttle message.
- `signUp` with email confirmation on returns no session → the store stays un-erroed and the UI shows
  its success panel (link-sent / redirect); the reset link lands on `/update-password` where the
  session is established before `updateUser`.
- Empty submit is blocked by the disabled button + guard; the `redirect` param is sanitised (spec 0001).

## 8. Cost & performance
No new infra — Supabase Auth (free tier), transactional reset email via the built-in mailer (TRD §2).
Profile hydration is deferred (`setTimeout(…,0)`) so first paint is not blocked; refresh events skip
redundant profile fetches. Auth-state bloat is a watch-item — keep `user_metadata` minimal (only
`full_name`) to avoid JWT/cookie growth.

## 9. Test plan
- **Functional:** register → signed in on `/home` with a fresh empty library; forgot-password →
  reset mail → `/update-password` sets a new password; wrong/duplicate/weak cases each assert the
  expected Kazakh string from `translateAuthError`.
- **Redirects:** signed-out `(app)` route → `/login?redirect=…`; signed-in `/login` → `/home`.
- **a11y:** Playwright + axe-core across all four `(auth)` screens — labels, `aria-invalid`/`describedby`
  wiring, the `role="alert"` error, the `role="status"` success, and the toggle `aria-pressed`.
- **Done** = all four flows work, every error is localised, and the axe suite passes on each screen.

## 10. Open questions
Email confirmation is currently effectively off (signup returns a session directly); if it is enabled,
the register screen's success/redirect copy should shift to a "check your email" state. Email delivery
depends on the Supabase SMTP configuration — out of scope here, tracked in ops.
